# ADR-004: Agent-to-PostgreSQL Connectivity

**Status:** Accepted

## Context

Each agent (running on a remote Linux host or on the central host itself) must insert rows into the PostgreSQL instance on the central host. PostgreSQL listens on port 5432. Two connectivity models were evaluated:

1. **Direct TCP connection:** The agent connects to PostgreSQL on the central host's network IP or hostname. PostgreSQL's `pg_hba.conf` must include an entry permitting connections from the agent host's IP address. The Postgres port is reachable from remote hosts on the local network.

2. **SSH tunnel:** The agent establishes an SSH tunnel to the central host (using the same SSH keys already set up for deployment) and connects to PostgreSQL via `localhost` on the tunnel. The Postgres port is not exposed on the network; all access goes through the SSH daemon.

Key contextual facts:

- SSH key exchange is already required between the central host and each remote host for agent deployment. The SSH infrastructure exists regardless of which database connectivity model is chosen.
- The deployment environment is described as a local network (1–5 machines). There is no mention of an untrusted network segment or internet exposure.
- The developer is solo on a prototype timeline. Operational simplicity is a priority.
- No TLS requirement for the database connection is stated in SPEC.md.

## Decision

Agents connect to PostgreSQL via a direct TCP connection to the central host's network IP or hostname. The database connection hostname is stored in the agent's configuration file. Credentials are supplied via environment variable or config file and are never stored in source code.

`pg_hba.conf` on the central host must be configured to allow connections from the agent host IP addresses using password authentication (md5 or scram-sha-256). The Postgres port (5432) should be firewalled to permit only the known agent host IPs.

## Consequences

### Easier

- The agent's database connection is a single `psycopg2.connect(...)` call with a hostname. No tunnel process to start, monitor, or restart.
- Startup and reconnect logic (FM-4 in ARCH.md §6) is handled entirely within the agent process; there is no external SSH subprocess to manage.
- Debugging connectivity is straightforward: `psql -h <central-host> -U <user> smart_disk_monitor` from the remote host confirms or denies access without tunnelling complexity.
- Standard PostgreSQL connection pooling or retry libraries work without modification.

### Harder

- The PostgreSQL port must be accessible from each remote host on the network. This requires a `pg_hba.conf` rule per agent host IP and a firewall rule permitting the connection. If agent host IPs change (e.g. DHCP), rules must be updated.
- The database password travels over the local network. On a trusted local network this is low risk, but it is not encrypted unless PostgreSQL's `sslmode` is configured. This is an accepted risk for v1 on a local network.
- Network exposure of the Postgres port is a larger attack surface than an SSH tunnel. Any host on the network that can reach port 5432 can attempt authentication. Firewall rules and strong passwords mitigate this.

## Alternatives Rejected

### SSH Tunnel

The agent runs `ssh -L 5432:localhost:5432 <central-host>` (or equivalent) to create a local tunnel, then connects to `localhost:5432`.

Advantages: the Postgres port is never exposed on the network; all database traffic is encrypted within the SSH session; no `pg_hba.conf` changes beyond localhost are needed.

Reasons rejected for v1:

- **Tunnel lifecycle management adds operational complexity.** The agent must start the SSH subprocess before the database connection, detect if the tunnel drops, restart it, and avoid port conflicts if multiple agents run on the same host. This is non-trivial to make reliable.
- **The SSH infrastructure already exists for deployment only.** Repurposing it for ongoing runtime connectivity couples the agent's operation to SSH availability in a way that is harder to debug than a failed TCP connection.
- **The threat model does not require it for v1.** The deployment environment is a local network. A firewall rule restricting port 5432 to known agent IPs provides sufficient isolation without the tunnel complexity.
- The SSH tunnel approach is the correct hardening step if this system is later deployed across an untrusted network or if the Postgres port must not be exposed at all. It should be reconsidered at that point.

### Unix Domain Socket (Central Host Agent Only)

For the agent running on the central host itself, connecting via Unix domain socket (`/var/run/postgresql/.s.PGSQL.5432`) avoids TCP entirely.

This is a valid optimisation for the local agent but cannot apply to remote agents. Using two different connectivity models (socket for local, TCP for remote) would make the agent code more complex and the deployment procedure inconsistent. A single TCP model works for all agent instances, including the local one.

Unix socket connectivity was rejected in favour of a uniform TCP approach across all agent deployments.

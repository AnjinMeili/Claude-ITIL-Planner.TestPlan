# ADR-001: Agent Implementation Language

**Status:** Accepted

## Context

The collection agent must run on each monitored Linux host (1–5 machines). Its core responsibilities are:

- Invoking `smartctl` as a subprocess with `sudo`
- Parsing JSON or text output from `smartctl`
- Connecting to PostgreSQL and inserting rows
- Running on a configurable interval via an internal scheduler

The developer is a solo engineer on a prototype timeline measured in days. Key forces:

- **Availability on target hosts without extra install.** The agent will be deployed to Linux machines that may not have non-standard runtimes pre-installed. Installing a runtime is acceptable but adds friction.
- **Subprocess invocation ergonomics.** The agent calls `smartctl` repeatedly, parses its stdout, and handles timeouts and non-zero exit codes. This is a first-class use case for the language, not an edge case.
- **PostgreSQL client library quality.** The agent must connect to Postgres, perform inserts, handle connection errors, and retry. The client library must be stable and well-documented.
- **Prototype speed.** With a days-long timeline, development velocity matters more than long-term operational properties.
- **Developer familiarity.** A solo developer's existing proficiency dominates on a short timeline.

Python 3 ships with or is trivially available on virtually all Linux distributions. Its `subprocess` module, `json` standard library, `psycopg2` (or `psycopg3`) Postgres adapter, and `schedule` or `threading` primitives cover every agent requirement with mature, well-documented libraries. Go is a strong operational choice but carries higher upfront cost for a prototype.

## Decision

The agent is implemented in Python 3. The web server is also implemented in Python 3 (Flask or equivalent), keeping the runtime dependency uniform across all components.

## Consequences

### Easier

- Subprocess invocation, JSON parsing, and Postgres connectivity are all covered by the standard library or widely-available packages with minimal boilerplate.
- Deployment is straightforward on Linux: Python 3 is either pre-installed or available via the system package manager (`apt`, `dnf`, `pacman`).
- A single language across agent and web server means the developer context-switches less and can share utility code (e.g. the `DeviceReading` data class or the DB connection helper).
- The prototype timeline is achievable; there is no compilation step and REPL-driven iteration is fast.

### Harder

- Python's GIL and startup overhead are irrelevant at this scale, but the agent is not a suitable base for high-frequency, low-latency collection if requirements change significantly.
- Packaging for deployment requires either a virtualenv per host, a bundled wheel archive, or acceptance that system Python has the required packages. This is solvable but requires a documented install procedure.
- Runtime errors surface at execution time rather than compile time. A statically typed language would catch certain classes of bug earlier.

## Alternatives Rejected

### Go

Go produces a single statically linked binary, which makes deployment trivial: copy one file, no runtime dependency. It handles subprocess invocation and Postgres (via `pgx`) well. However:

- The development overhead of writing idiomatic Go (error handling verbosity, lack of a REPL, compilation cycle) makes a days-long prototype harder to execute for a developer not already fluent in Go.
- The operational benefits (single binary, low memory, fast startup) are not differentiating at a scale of 1–5 hosts.
- Go was rejected in favour of Python for this prototype. It remains the stronger choice if the agent is later productionised or distributed to many hosts.

### Bash / Shell Script

Shell scripts can invoke `smartctl` and use `psql` to write to Postgres. For a prototype this is tempting. However:

- Parsing JSON from `smartctl -j` output in shell requires `jq` as an additional dependency.
- Error handling (subprocess timeouts, per-device retry, structured logging) is significantly more brittle in shell than in Python.
- The data normalisation step (mapping raw `smartctl` output to the internal `DeviceReading` schema) would be fragile to maintain as output formats vary across device types.
- Shell was rejected as insufficiently maintainable for the complexity of flag detection and output parsing.

# ADR-003: Web Rendering Approach

**Status:** Accepted

## Context

The web application has one use case in v1: display a single HTML table showing all devices across all machines, with the most recent SMART reading per device (UC-05). The confirmed v1 constraints are:

- No authentication (explicit constraint)
- No alerting
- Display-only: no user input, no write operations from the browser
- Solo developer, prototype timeline

The rendering approach determines how the web server and browser collaborate to produce the HTML table. Two approaches were evaluated:

1. **Server-side rendered HTML:** The web server queries PostgreSQL on each HTTP request and returns a complete HTML page. The browser receives and displays HTML. No JavaScript is required.

2. **JSON API + JavaScript frontend:** The web server exposes a `/api/devices` endpoint returning JSON. A JavaScript frontend (vanilla JS, or a framework like Vue or React) fetches the JSON and renders the table in the browser.

## Decision

The web server uses server-side HTML rendering. On each GET request to `/`, the web server queries PostgreSQL for the latest reading per `(host_id, device_path)`, renders a complete HTML table using a server-side template (Jinja2), and returns the full HTML document. No JavaScript is required for the table to be displayed.

## Consequences

### Easier

- No build step, no JavaScript bundler, no `node_modules`. The frontend is a single Jinja2 template.
- The full page is visible immediately on load, even in browsers with JavaScript disabled.
- Debugging is simpler: the rendered HTML is inspectable directly from the browser's view-source.
- The query, rendering, and error handling are all in one place (the Flask route handler), reducing the number of components that can fail independently.
- For the expected data volume (≤5 hosts × ≤20 devices), a full-page render on each request is well within acceptable latency. No caching layer is needed.

### Harder

- Adding interactive features (client-side sorting, column filtering, auto-refresh without full page reload) requires either adding JavaScript later or restructuring toward a hybrid approach. The server-side approach is not a dead end for these features, but they would require more work than if a JSON API already existed.
- If the data volume grows significantly (hundreds of devices, high request rate), full-page renders on each request become less efficient than a cached JSON API with incremental updates. This is not a concern at v1 scale.
- The page goes fully stale between manual refreshes. There is no auto-refresh in v1; the operator must reload the page to see updated data. (A `<meta http-equiv="refresh">` tag can be added without changing the architecture if desired.)

## Alternatives Rejected

### JSON API + JavaScript Frontend

A `/api/devices` endpoint returns JSON; a JavaScript frontend renders the table.

This approach provides a clean separation between data and presentation and would make future features (auto-refresh via polling, client-side sort/filter, row highlighting for FAILED status) easier to add.

However, for v1's display-only requirement, the additional complexity is not justified:

- A JavaScript frontend introduces a build toolchain concern (even with vanilla JS, the separation adds files and indirection).
- Two components can fail independently: the API endpoint and the frontend rendering logic. Debugging a blank table requires checking both.
- The JSON API layer is not required by any v1 use case. Adding it now would be designing beyond SPEC.md.

If v1 proves useful and interactive features are requested, a JSON API can be added alongside the existing server-side route without requiring a full rewrite of the rendering approach.

### Static File Generation (Cron-Generated HTML)

A separate process (or a post-collection hook in the agent) generates a static HTML file after each collection cycle. The web server simply serves the static file.

This eliminates the database query on each HTTP request entirely. However:

- It introduces coordination between the agent and the web server (the agent must write to a location the web server can read, or trigger a regeneration script).
- The static file can be stale by up to one collection interval. Server-side rendering on demand gives the operator the freshest data available at the time of the request.
- The added coordination complexity is not warranted at this scale.

Static generation was rejected as unnecessarily complex for v1.

# ADR-002: smartctl Flag Detection Strategy

**Status:** Accepted

## Context

`smartctl` requires different invocation flags depending on the device type:

- SATA drives: typically no flag, or `--device=ata`
- NVMe drives: `--device=nvme`
- USB-attached drives with SAT passthrough: `--device=sat` or `--device=sat,12`
- RAID controllers: vendor-specific flags (e.g. `--device=megaraid,N`)

A device that receives the wrong flag may return an error, return partial data, or (in some firmware cases) appear to succeed but return meaningless values. The requirement AC-03-03 explicitly prohibits hardcoded device maps: the system must not rely on a static lookup of device paths to flag strings.

Three strategies were evaluated:

1. **Scan-and-probe:** Enumerate devices via `smartctl --scan` or `lsblk`, then attempt a fixed set of candidate flags in order (no flag → `--device=auto` → `--device=ata` → `--device=nvme` → ...) until one returns exit code 0.

2. **JSON info probe:** Run `smartctl -i -j <device>` (the info query in JSON mode) for each device first. Parse the `device.type` field from the JSON output to determine the correct flag, then run the full health query with that flag.

3. **Kernel udev/sysfs inspection:** Read `/sys/block/<dev>/device/` attributes (e.g. `transport`, `vendor`) and `/run/udev/data/` to determine the bus type before invoking `smartctl` at all.

The Flag Detector must not hang on unresponsive devices (FM-1 in ARCH.md §6). All strategies must be wrapped with a per-device subprocess timeout regardless of which is chosen.

## Decision

Flag detection uses the JSON info probe strategy (option 2). For each enumerated device, the Flag Detector runs:

```
sudo smartctl -i -j <device_path>
```

It parses the `device.type` field from the JSON output to classify the device and select the appropriate flags for the subsequent health query. If the info probe fails or returns an unrecognised type, the Flag Detector falls back to no flag (default behaviour) and marks the type as `"unknown"`.

## Consequences

### Easier

- `smartctl`'s own device classification logic is used, which handles the widest range of device types correctly without the agent needing knowledge of device-type heuristics.
- The JSON output format is stable across `smartctl` versions and is machine-parseable without fragile string matching.
- A single probe per device (the info query) is sufficient to determine flags; no iterative retry loop is needed for the common case.
- The strategy naturally handles NVMe, SATA, USB passthrough, and many RAID configurations because `smartctl -i` is designed to identify device type before issuing health queries.

### Harder

- Requires `smartctl` to support the `-j` (JSON output) flag, which was introduced in smartmontools 7.0 (released 2019). Hosts running older distributions with smartmontools 6.x will need to either upgrade or receive a text-parsing fallback.
- Two `smartctl` invocations are required per device per cycle (one info probe, one health query) rather than one. For 1–5 hosts with ≤20 devices each, this is not a performance concern but does double the subprocess call count.
- If `smartctl -i -j` returns exit code 0 but the `device.type` field is absent or null (possible for exotic or unsupported devices), the fallback to no flag may produce incorrect results.

## Alternatives Rejected

### Scan-and-Probe (Option 1)

Enumerate devices, then iterate through a ranked list of candidate flags until `smartctl` exits 0.

This strategy has two fundamental weaknesses:

- **Flag order matters and is not universally correct.** A USB drive with SAT passthrough may return exit code 0 with `--device=auto` but produce incorrect health data. Exit code 0 does not mean the right flag was used; it means `smartctl` did not hard-fail.
- **Error codes are not reliable across firmware.** Some devices return exit code 0 with partial output; others return non-zero for conditions that are not actually errors (e.g. smartctl exit bitmask bit 1 = "device open failed" vs bit 2 = "SMART command failed"). Interpreting these reliably requires reconstructing the same classification logic that `smartctl` itself already performs.
- **The probe loop is slow for unresponsive devices.** Each candidate flag attempt may hang until the per-device timeout fires, multiplying the worst-case collection time by the number of candidate flags.

Scan-and-probe was rejected as fragile and prone to false positives.

### Kernel udev/sysfs Inspection (Option 3)

Read `/sys/block/<dev>/device/` or `/run/udev/data/` attributes to determine bus type before any `smartctl` invocation.

This approach has significant brittleness concerns:

- The sysfs attribute paths and their values vary across kernel versions, distributions, and device drivers. An attribute that reliably indicates "USB mass storage" on one kernel may be absent or differently named on another.
- USB-attached drives with SAT passthrough are common and are particularly difficult to classify via sysfs because the USB subsystem presents the device as a SCSI device regardless of the underlying protocol.
- RAID controller enumeration via sysfs requires knowledge of each controller's sysfs layout, which varies by vendor and driver — reintroducing a form of hardcoded device map.
- This strategy requires the agent to maintain kernel-interface knowledge that `smartctl` already encapsulates. Any kernel upgrade that changes sysfs layout would silently break flag detection without a `smartctl` version change.

Sysfs inspection was rejected as too brittle across kernel versions and distributions.
